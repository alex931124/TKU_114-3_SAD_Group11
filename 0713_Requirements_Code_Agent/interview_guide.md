## C. 角色 1：宿舍學生

| 欄位 | 內容 |
| --- | --- |
| 受訪角色 | 宿舍學生 |
| 角色與系統關係 | 主要向系統提交宿舍報修請求，並查詢進度 |
| 訪談目的 | 確認學生報修流程、痛點與必要資料 |

| 題號 | 訪談問題 | 想確認的需求或風險 |
| --- | --- | --- |
| 1 | 你平常是怎麼向宿舍管理員或維修人員回報問題？ | 了解目前作業方式與資訊落點 |
| 2 | 當你提交報修後，通常多久能知道是否有人收到？ | 確認狀態回饋與通知需求 |
| 3 | 你會希望系統在報修單中要求填哪些內容？哪些內容是最重要的？ | 找出必填欄位與資料需求 |
| 4 | 如果報修問題重複發生或多次回報，你會怎麼判斷是否已提交？ | 檢查重複通報或重複處理風險 |
| 5 | 如果宿舍管理員尚未派工，現在你會怎麼知道進度？ | 確認待處理狀態與在場的回饋 |
| 6 | 你認為系統最重要的成功標準是什麼？ | 確認可驗收成果 |

## D. 角色 2：宿舍管理員

| 欄位 | 內容 |
| --- | --- |
| 受訪角色 | 宿舍管理員 |
| 角色與系統關係 | 查看報修清單、分派維修人員、更新處理狀態 |
| 訪談目的 | 確認派工責任、狀態轉換與角色權限需求 |

| 題號 | 訪談問題 | 想確認的需求或風險 |
| --- | --- | --- |
| 1 | 你目前是如何檢視宿舍學生的報修需求？有哪些常見處理瓶頸？ | 了解現況流程與痛點 |
| 2 | 如果有多個維修人員，現在如何分派任務？是否有固定排班或區域分工？ | 釐清派工規則 |
| 3 | 請問「待派工」、「派工中」、「已完成」等狀態是否需要明確區分？ | 判斷系統狀態模型 |
| 4 | 當報修單缺少必要資料時，你會怎麼處理？ | 找出例外情況與欄位驗證需求 |
| 5 | 你需要查看哪些資訊才能確定維修人員是否接單？ | 確認派工紀錄與回報資料 |
| 6 | 若系統要支援你們的作業，什麼樣的成功標準最關鍵？ | 評估本期切片是否能驗收 |

## E. 角色 3：維修人員

| 欄位 | 內容 |
| --- | --- |
| 受訪角色 | 維修人員 |
| 角色與系統關係 | 接受派工並回報完成情況 |
| 訪談目的 | 確認維修完成流程與回報需求 |

| 題號 | 訪談問題 | 想確認的需求或風險 |
| --- | --- | --- |
| 1 | 你現在收到派工通知的方式是什麼？ | 了解既有通知方式 |
| 2 | 完成維修後，通常要回報哪些資訊？ | 確認完成回報的最小資料 |
| 3 | 如果工項超出預期、需要額外材料或無法立即完成，現在怎麼處理？ | 找出例外情況與流程限制 |
| 4 | 你希望在清單上看到什麼資訊，才能快速判斷優先順序？ | 確認清單欄位需求 |
| 5 | 你會希望系統記錄哪些時間點？例如接單、開始、完成 | 訊息化維修流程 |
| 6 | 如果這個系統只做第一個可操作切片，最重要的更新資訊是什麼？ | 聚焦本期範圍 |
