# 候選需求清單（Candidate Requirements List）

| 需求編號 | 需求描述 | 類型 | 來源編號 | 本期是否納入 | 理由 |
| --- | --- | --- | --- | --- | --- |
| FR-01 | 宿舍學生應能在一個統一報修入口輸入宿舍、房號、設備類型與問題描述，並送出報修申請。 | 功能性需求（FR） | INT-STU-01、INT-STU-03 | 是 | 與學生提交報修單核心任務直接對應 |
| FR-02 | 系統應在學生提交報修後，顯示案件初始狀態與案件編號，讓學生知道申請已受理。 | 功能性需求（FR） | INT-STU-01、INT-STU-02 | 是 | 這是學生最直接的資訊回饋 |
| FR-03 | 系統應讓宿舍管理員能查看待處理報修清單，並將案件派給維修人員。 | 功能性需求（FR） | INT-ADM-01、INT-ADM-02 | 是 | 對應主管理員的派工需求 |
| FR-04 | 系統應讓管理員更新案件狀態，並保留狀態變更紀錄。 | 功能性需求（FR） | INT-ADM-02、OBS-ADM-01 | 是 | 可支援第一個切片的狀態追蹤 |
| FR-05 | 系統應讓維修人員查看已派工任務清單，並更新完成狀態與備註。 | 功能性需求（FR） | INT-MNT-01、INT-MNT-02 | 是 | 對應完成回報的核心流程 |
| NFR-01 | 報修表單的必填欄位應清楚標示，避免填寫資料缺漏。 | 非功能性需求（NFR） | INT-ADM-03 | 是 | 可用於表單驗證與操作流暢度 |
| NFR-02 | 報修清單與狀態資訊應能清楚呈現，以降低學生與管理員反覆詢問的成本。 | 非功能性需求（NFR） | INT-STU-02、OBS-ADM-01 | 是 | 對應狀態可視性與使用方便性 |
| CON-01 | 本期原型只處理報修、派工與狀態追蹤，不做正式單一登入。 | 限制條件（constraint） | DOC-CHA-01 | 是 | 由章程定義範圍，避免代理自動擴大功能 |
| CON-02 | 本期不做實體維修零件採購、手機原生 App 與正式部署。 | 限制條件（constraint） | DOC-CHA-01、DOC-OUT-01 | 否 | 本期範圍外，若納入會使原型過大 |
| FR-06 | 系統應辨識重複報修需求，避免相同問題被重複送出。 | 功能性需求（FR） | INT-NEW-01 | 否 | 目前尚不足以支撐本期切片，後續可再補充 |
