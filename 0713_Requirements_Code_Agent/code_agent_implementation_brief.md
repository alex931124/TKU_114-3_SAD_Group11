# 程式碼代理實作任務書（Code Agent Implementation Brief）

## 0. 工具選擇

| 項目 | 內容 |
| --- | --- |
| 本組使用的程式碼代理（code agent） | GitHub Copilot Chat / IDE Agent |
| 使用介面 | VS Code 整合開發環境（IDE） |
| 選擇原因 | 可邊讀檔案邊改程式、產出原型且保留人工修正紀錄 |
| 是否需要登入或授權 | 是，需對應工作區與 Copilot 授權 |
| 是否會上傳專案內容到外部服務 | 會，依 IDE / GitHub 服務設定而定 |
| 小組如何避免外送不該外送的資料 | 只在工作區內提交需求文件與假資料，避免上傳真實帳號或正式資料 |

## 1. 專案背景

本組專案是「校園宿舍報修、派工與維修追蹤系統」。目前宿舍學生多半以 LINE、電話或口頭方式回報問題，管理員與維修人員則需手動整理資訊。這次目標是先用低擬真網頁原型，驗證「學生報修、管理員派工、維修人員回報」三個核心流程。

## 2. 目標角色

| 角色 | 主要任務 | 主要痛點 |
| --- | --- | --- |
| 宿舍學生 | 送出報修需求並查看狀態 | 目前無統一提交入口與狀態回饋 |
| 宿舍管理員 | 查看待處理清單、分派維修人員 | 資訊分散、清單不集中 |
| 維修人員 | 查看派工任務、完成維修並回報 | 缺少統一任務看板與狀態紀錄 |

## 3. 第一個可操作切片（First Working Slice）

本次只做：「宿舍學生送出報修單，宿舍管理員在清單中看到待處理案件，並將案件派給維修人員」。

## 4. 本期範圍內項目（In-Scope）

1. 報修單表單
2. 待處理清單
3. 狀態標籤
4. 角色切換（學生 / 管理員 / 維修人員）
5. 假資料與前端動態展示

## 5. 本期範圍外項目（Out-of-Scope）

1. 正式單一登入
2. 正式資料庫
3. 真實金流
4. 手機原生 App
5. 維修零件採購流程

## 6. 本次要實作的需求

| 需求編號 | 來源編號 | 需求描述 | 驗收條件 | 對應畫面或操作 |
| --- | --- | --- | --- | --- |
| FR-01 | INT-STU-01、INT-STU-03 | 宿舍學生應能輸入宿舍、房號、設備類型、問題描述並送出報修單。 | 送出後，表單資料會出現在管理員清單中。 | 報修表單、待處理清單 |
| FR-02 | INT-STU-02 | 系統應在送出後顯示案件狀態為「已提交」。 | 送出後可看到已提交狀態。 | 報修成功提示 |
| FR-03 | INT-ADM-01、INT-ADM-02 | 系統應讓管理員查看待處理報修清單並點選派工。 | 點選派工後，案件狀態更新為「已派工」。 | 管理員清單、派工操作 |
| FR-04 | INT-MNT-01、INT-MNT-02 | 系統應讓維修人員查看已派工任務並更新完成狀態與備註。 | 完成後，狀態更新為「已完成」。 | 維修人員清單、完成回報 |
| NFR-01 | INT-ADM-03 | 表單必填欄位應明確標示，空白時不能送出。 | 房號、設備類型、問題描述為必填。 | 報修表單驗證 |

## 7. 技術限制（Technical Constraints）

| 限制項目 | 本組設定 |
| --- | --- |
| 前端技術 | HTML、CSS、JavaScript |
| 後端技術 | 不使用正式後端 |
| 資料儲存 | 使用前端假資料 / localStorage |
| 是否需要正式部署 | 否 |
| 是否允許外部套件 | 只可使用最少必要外部套件 |
| 是否允許正式登入 | 否 |

## 8. 程式碼代理不可自行決定的事項

1. 不得自行新增正式登入角色或帳號系統。
2. 不得自行加入金流、採購或校務整合。
3. 不得把本期的三個核心任務擴大為完整宿舍管理系統。

## 9. 若代理想加入，必須拒絕的功能

| 功能 | 拒絕理由 | 對應範圍外項目 |
| --- | --- | --- |
| 正式單一登入 | 本期沒有這個需求，也不在章程範圍內 | 正式校內單一登入 |
| 真實資料庫與後端串接 | 原型只使用假資料，且不含正式部署 | 正式資料庫 |
| 手機原生 App | 會擴大開發範圍且不符合本期切片 | 手機原生 App |

```text
請依照以下系統分析與設計資料，建立第一個可操作切片。

專案背景：
校園宿舍報修、派工與維修追蹤系統，重點是讓宿舍學生提交報修需求，宿舍管理員查看待處理清單並派工，維修人員完成後回報結果。

1. 宿舍學生：提交修繕需求
2. 宿舍管理員：查看待處理清單並派工

FR-01, FR-02, FR-03, FR-04, NFR-01

2. 管理員可在清單看到新增案件

4. 維修人員可更新完成狀態與備註
5. 必填欄位空白時不可送出

不可捏造事項：
1. 不得新增正式登入角色
2. 不得加入金流、採購流程
3. 不得擴大為完整校園宿舍系統

請先產生實作計畫，確認後再開始建立或修改檔案。
```
