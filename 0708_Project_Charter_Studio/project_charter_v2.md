## C. 專案章程第 2 版（Project Charter v2）

### 1. 專案題目（Project Title）

校園宿舍報修、派工與維修追蹤系統

### 2. 目標使用者（Target Users）

| 角色       | 主要任務                                 | 主要痛點                                                 |
| ---------- | ---------------------------------------- | -------------------------------------------------------- |
| 宿舍學生   | 提交報修單、查詢維修進度                 | 目前多靠電話、訊息或紙本回報，無法確認維修進度與派工情況 |
| 宿舍管理員 | 查看報修需求、分派維修人員、追蹤維修進度 | 報修資訊分散，難以快速分派與追蹤處理狀態                 |
| 維修人員   | 接受派工、完成維修並回報結果             | 缺少統一維修任務清單與狀態更新機制                       |
| 系統管理員 | 管理帳號、角色與權限                     | 缺少集中式角色管理與帳號權限控管                         |

### 3. 問題陳述（Problem Statement）

宿舍學生與宿舍管理員在校園宿舍報修流程中，常因使用電話、訊息或紙本紀錄而導致資訊分散、處理狀態不清楚、派工效率低落，維修人員也缺少統一回報渠道。此系統希望整合報修申請、維修派工與維修狀態追蹤，讓報修流程更透明、分工更清楚，並提高維修任務完成的可追蹤性。本期聚焦於報修提交、派工管理與維修狀態追蹤，不處理實體零件採購、真實金流、正式校內單一登入與正式部署。

### 4. 專案目標（Project Goal）

設計一個可支援校園宿舍報修申請、維修派工與狀態追蹤的系統，讓宿舍學生、宿舍管理員與維修人員能更有效率地處理維修需求、了解任務狀態，降低資料分散與溝通延遲。

### 5. 利害關係人摘要（Stakeholder Summary）

宿舍學生是主要使用者，需求是快速提交報修單並查看處理進度；宿舍管理員需要一個集中式系統來檢視報修列表、分派維修人員並更新處理狀態；維修人員需要明確的派工任務與完成回報流程；系統管理員則負責帳號、角色與權限管理。其他受影響角色包括宿舍行政人員與學校資訊處，對後續系統擴充與資料整合提出安全與規範要求。

### 6. 範圍內項目（In-Scope）

1. 報修單建立與編輯
2. 報修狀態查詢與進度追蹤
3. 維修派工與派工人員狀態更新
4. 維修完成回報與關閉流程
5. 角色與基本權限管理

### 7. 範圍外項目（Out-of-Scope）

1. 實體維修零件採購流程
2. 真實金流付款流程
3. 正式校內單一登入整合
4. 正式部署到學校伺服器
5. 手機原生 App 開發

### 8. 原型核心任務（Core Tasks for Prototype）

| 任務                  | 主要角色   | 任務起點           | 任務終點                     | 初步驗收標準                                 |
| --------------------- | ---------- | ------------------ | ---------------------------- | -------------------------------------------- |
| 1. 提交報修單         | 宿舍學生   | 進入報修單提交頁面 | 成功建立報修單並顯示報修狀態 | 系統顯示報修內容、位置、問題類型與初始狀態   |
| 2. 派工與更新維修狀態 | 宿舍管理員 | 進入待處理報修清單 | 分派維修人員並更新任務狀態   | 系統顯示派工結果與當前狀態（待派工、進行中） |
| 3. 完成維修並回報結果 | 維修人員   | 進入已派工任務清單 | 完成維修並回報結果           | 系統更新完成狀態並顯示完成時間與備註         |

### 9. 初步成功標準（Initial Success Criteria）

1. 宿舍學生可以成功提交報修單並查詢維修進度。
2. 宿舍管理員可以順利分派維修人員並更新任務狀態。
3. 維修人員可以完成維修並回報維修結果。
4. 原型能展示從報修提交到派工再到完成回報的三個核心流程。
5. 範圍與功能能對應 5 週內可完成的低擬真原型。

### 10. 主要風險與應對方式（Major Risks and Responses）

| 風險                                 | 影響 | 可能性 | 應對方式                                                                 |
| ------------------------------------ | ---- | ------ | ------------------------------------------------------------------------ |
| 範圍太大，導致無法在 5 週內完成原型  | 高   | 高     | 將採購、金流、正式登入與正式部署移到範圍外，專注報修、派工與狀態追蹤。   |
| 權限與派工流程定義不清，導致原型混亂 | 中   | 中     | 先定義最小角色與簡化狀態轉換流程，限縮為單一維修人員派工與基本角色管理。 |
| 組員對工具不熟悉，影響原型品質       | 中   | 中     | 先完成低擬真原型並統一使用簡單工具，避免過度開發。                       |
